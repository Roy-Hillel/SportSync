# Plan 3.2: Provider Registration and Env Var Wiring

## Objective

Register the new `ApiFootballProvider` in the provider registry so `SPORTS_PROVIDER=api-football` loads it, update `.env.local` to activate the new provider, and update `.env.local.example` to document the new env vars.

**Requirements covered:** PROV-06, PROV-07

**Depends on:** Plan 3.1 (the `getApiFootballProvider` export must exist)

## Context

**Provider registry** (`src/lib/providers/index.ts`):
```typescript
import type { SportsDataProvider } from "./types";
import { getSportRadarProvider } from "./sportradar";

export type { SportsDataProvider, ProviderEntity, ProviderEvent, EntityType, EventStatus } from "./types";

const PROVIDERS: Record<string, () => SportsDataProvider> = {
  sportradar: getSportRadarProvider,
};

export function getActiveProvider(): SportsDataProvider {
  const name = process.env.SPORTS_PROVIDER ?? "sportradar";
  const factory = PROVIDERS[name];
  if (!factory) {
    throw new Error(
      `Unknown SPORTS_PROVIDER "${name}". Available: ${Object.keys(PROVIDERS).join(", ")}`
    );
  }
  return factory();
}
```

The registry already handles unknown provider names with a clear error. Adding a new provider is just: import the factory function, add one entry to the `PROVIDERS` map.

**Current `.env.local.example`** does not include `API_FOOTBALL_KEY` or `SPORTS_PROVIDER`. It has:
- `DATABASE_URL`, `AUTH_SECRET`, `AUTH_GOOGLE_ID`, `AUTH_GOOGLE_SECRET`, `SPORTRADAR_API_KEY`, `CRON_SECRET`, `NEXTAUTH_URL`

**Current `.env.local`** has `API_FOOTBALL_KEY` already set (value present). Does NOT have `SPORTS_PROVIDER` set (defaults to `sportradar`).

**Vercel dashboard** needs `API_FOOTBALL_KEY` and `SPORTS_PROVIDER=api-football` added manually -- this is a human action for later (Phase 6 cutover).
