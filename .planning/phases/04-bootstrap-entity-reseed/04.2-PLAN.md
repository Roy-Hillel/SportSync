## Context

### Why deletion is required before re-seed

The `subscribable_entities` upsert conflict target is `(provider_id, provider)`. A row with `provider = 'sportradar'` and `provider_id = 'sr:competition:7'` will never conflict with an API-Football row at `provider = 'api-football'` and `provider_id = '2'` — they are completely separate rows. Without deletion, old SportRadar rows remain indefinitely, polluting search results with stale entities that have non-functional IDs.

### Current state of the table (from ENGINEERING_NOTES.md)
- 1,265 competition rows — all `provider = 'sportradar'`
- ~10,176 team rows — all `provider = 'sportradar'`
- 999 teams with `(W)` suffix

After this plan, the target state is:
- 0 rows with `provider = 'sportradar'`
- N competitions with `provider = 'api-football'` (expect: all currently active leagues from `/leagues?current=true`)
- M teams with `provider = 'api-football'` (from priority countries + international competitions)

### Rate limit math

The bootstrap fetches:
- 1 request for `GET /leagues` (all competitions)
- 1 request per priority competition for `GET /teams?league=<id>&season=<year>`

Estimate ~50 priority competitions (country filter + explicit IDs) → ~51 total requests. Well within the 7,500 req/day API-Football Pro limit and the 300 req/min rate limit.

## Rollback Plan

There is no automated rollback. The SportRadar data was seeded from SportRadar API calls which cannot be replayed (quota was used at initial seed time and the trial is no longer the active provider).

Risk assessment: LOW. The old SportRadar rows are stale — they point to IDs that the current provider no longer uses. The app with `SPORTS_PROVIDER=api-football` cannot function with SportRadar entity IDs in `subscribable_entities`. Deletion is required for correct operation.

If bootstrap fails catastrophically mid-run, re-run the script — the upsert is idempotent.
